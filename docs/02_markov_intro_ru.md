# Intro к ноутбуку 02 — Марковская модель рисков

## 1) Что делает ноутбук 02

Ноутбук [notebooks/02_Markov_ru.ipynb](notebooks/02_Markov_ru.ipynb) переводит выводы EDA в рабочую риск-модель, которая нужна для оптимизации лимитов.

Цель: для каждого клиента получить риск- и value-параметры, которые пойдут в оптимизатор:

EV_i = 40 - PD_i * LGD_i * EAD_i

Выходной артефакт ноутбука: [data/df_risk.csv](data/df_risk.csv).

Он добавляет к данным поля:
- pd_monthly
- pd_annual
- lgd
- ead
- el_per_increase
- ev_per_increase

Функции вынесены в модуль [src/markov_risk.py](src/markov_risk.py).

## 2) Как устроена модель

### 2.1 Состояния риска

Используются 4 состояния:
- Prime
- Near-Prime
- Sub-Prime
- Default (поглощающее состояние)

### 2.2 Матрица переходов

Месячная матрица переходов T задана в [src/markov_risk.py](src/markov_risk.py):
- Prime -> Default: 0.5% в месяц
- Near-Prime -> Default: 1.5% в месяц
- Sub-Prime -> Default: 5.0% в месяц
- Default -> Default: 100%

Сценарные наборы калибровок в коде:
- `T_soft`: Prime 0.35% / Near-Prime 1.20% / Sub-Prime 4.00%
- `T_base`: Prime 0.50% / Near-Prime 1.50% / Sub-Prime 5.00%
- `T_conservative`: Prime 0.75% / Near-Prime 2.20% / Sub-Prime 7.00%

Публичные reference-источники для baseline:
- FRED `DRCCLACBS` (Delinquency Rate on Credit Card Loans): https://fred.stlouisfed.org/series/DRCCLACBS
- FRED `CORCCACBS` (Charge-Off Rate on Credit Card Loans): https://fred.stlouisfed.org/series/CORCCACBS

Важно: эти источники используются как benchmark-якорь уровня кредитного стресса в экономике.
Матрицы `soft/base/conservative` являются портфельно-адаптированной калибровкой, а не прямой one-to-one оценкой из внешних рядов.

Почему выбраны именно эти значения:
- В исходном датасете нет наблюдаемой помесячной истории дефолтов и переходов состояний, поэтому прямую эмпирическую оценку сделать нельзя.
- Нужен baseline, который соблюдает монотонность риска: Prime < Near-Prime < Sub-Prime.
- Уровни 0.5% / 1.5% / 5.0% находятся в реалистичном порядке величин для необеспеченного потребительского портфеля и дают правдоподобный годовой риск после многошаговых переходов.
- Такой baseline не делает задачу тривиальной: часть Sub-Prime становится убыточной, то есть появляется реальная граница для оптимизатора.
- Default = 100% в строке Default отражает стандартное допущение марковской цепи: состояние дефолта поглощающее в рамках рассматриваемого горизонта.

Как читать эти числа правильно:
- Это не "истинные" PD из данных, а калибровочные гипотезы для сценарного моделирования.
- Они проверяются анализом чувствительности (смещение PD на +/-2 п.п.) и при появлении исторических данных должны быть переоценены статистически.

В модели есть и переходы между «живыми» состояниями (апгрейд/даунгрейд), поэтому годовая PD не равна 12 * monthly PD.

### 2.3 Годовая PD

Годовая PD считается через T^12 как вероятность попасть в Default за 12 шагов:
- Prime: 10.6%
- Near-Prime: 19.5%
- Sub-Prime: 40.4%

Откуда именно эти числа:
- Берем матрицу переходов T из [src/markov_risk.py](src/markov_risk.py).
- Возводим ее в 12-ю степень: T_12 = T^12.
- Смотрим 4-й столбец (Default) для стартовых состояний Prime/Near-Prime/Sub-Prime:
	- PD_annual(Prime) = T_12[Prime, Default]
	- PD_annual(Near-Prime) = T_12[Near-Prime, Default]
	- PD_annual(Sub-Prime) = T_12[Sub-Prime, Default]

Почему это важно:
- Это вероятность дойти до дефолта по всем траекториям за 12 месяцев, а не только "дефолт за один шаг".
- Поэтому PD за год не равна просто 12 * PD_monthly и не равна формуле 1 - (1 - PD_monthly)^12, если есть миграции между состояниями риска.

Проверка на интуицию (без миграций, только для сравнения):
- Prime: 1 - (1 - 0.005)^12 = 5.8% (ниже марковских 10.6%)
- Near-Prime: 1 - (1 - 0.015)^12 = 16.6% (ниже марковских 19.5%)
- Sub-Prime: 1 - (1 - 0.05)^12 = 46.0% (выше марковских 40.4%)

Интерпретация:
- Для Prime/Near-Prime миграции вниз повышают риск относительно "фиксированного" состояния.
- Для Sub-Prime возможные апгрейды частично снижают риск относительно сценария "всегда Sub-Prime".

Итоговые годовые PD отличаются от «наивной» оценки из одной только месячной PD, потому что модель учитывает многошаговые траектории и переходы между состояниями риска.

### 2.4 Экономика одного увеличения

Параметры в ноутбуке:
- Profit per increase = 40
- LGD = 55%
- EAD = 50% от initial_loan

Затем:
- EL per increase = PD_monthly * LGD * EAD
- EV per increase = 40 - EL per increase

## 3) Почему получились именно такие результаты

### 3.1 По уровням риска

Результаты:
- Prime: avg EV/inc = 36.2, EV>0 = 100%
- Near-Prime: avg EV/inc = 28.7, EV>0 = 100%
- Sub-Prime: avg EV/inc = 2.0, EV>0 = 53%

Причина:
- У Prime и Near-Prime месячная PD достаточно низкая, поэтому даже при EAD=50% ожидаемый убыток ниже 40 почти у всех клиентов.
- У Sub-Prime месячная PD = 5%. На больших initial_loan ожидаемый убыток EL становится выше 40, и EV уходит в минус.

### 3.2 Почему 3,463 клиента убыточны

Убыточные клиенты сосредоточены в Sub-Prime с более крупными кредитами:
- высокий PD
- высокий EAD (пропорционален initial_loan)
- фиксированная прибыль 40

Из-за этого EL > 40 для части портфеля, и получаем 3,463 клиентов с EV<=0.

### 3.3 Почему 88.5% клиентов все же выгодны

Большая часть портфеля находится в Prime/Near-Prime или имеет не слишком высокий EAD. Поэтому у 26,537 из 30,000 клиентов EV остается положительным.

## 4) Использовали ли выводы из EDA

Да, напрямую.

Использованы ключевые выводы из [notebooks/01_EDA.ipynb](notebooks/01_EDA.ipynb) и [notebooks/01_EDA_ru.ipynb](notebooks/01_EDA_ru.ipynb):

1. Историческое назначение увеличений случайно (A4), поэтому не строим «модель подражания» прошлым решениям.
2. Риск кодируем через on_time_pct и risk_state, а не через historical received.
3. Поле total_profit в исходных данных синтетическое (A6), поэтому в objective используется бизнес-формула 40 на увеличение и риск-коррекция через PD/LGD/EAD.
4. Подготовленные в EDA признаки из [data/df_features.csv](data/df_features.csv) стали входом ноутбука 02.

Итог: ноутбук 02 является логическим продолжением EDA, а не отдельной «изолированной» моделью.
