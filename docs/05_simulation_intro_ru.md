# Notebook 05: Монте-Карло симуляция стратегий (RU)

## 1) Цель

Notebook 05 нужен для проверки устойчивости стратегии, полученной в Notebook 04, уже не в детерминированной постановке, а в стохастической.

Основные цели блока:
- проверить, как ведет себя портфельный NPV при случайных реализациях принятия оффера и дефолтов;
- получить распределение NPV по сценариям `base / stress / benign`;
- оценить downside-риск через `VaR95` и `CVaR95`;
- провести sensitivity-анализ по ключевым драйверам `p_accept` и `PD`;
- подготовить артефакты для финального narrative и выводов по устойчивости стратегии.

## 2) Что делали

### 2.1 Входные данные

В симуляцию подаются результаты предыдущих этапов:
- [data/df_demand.csv](data/df_demand.csv): вероятности принятия оффера и риск-параметры;
- [data/allocation_recommended.csv](data/allocation_recommended.csv): рекомендованная аллокация `lp_round` из Notebook 04;
- [data/optimization_summary.csv](data/optimization_summary.csv): детерминированные KPI оптимизации.

### 2.2 Логика симуляции

В [src/simulation.py](src/simulation.py) реализована следующая схема:

1. Для каждого сценария собирается таблица клиентов с:
- `p_accept_scenario`,
- `pd_annual`,
- `ead`,
- `x_recommended`.

2. Внутри каждой симуляции:
- число принятий моделируется как `Binomial(offers, p_accept)`;
- число дефолтов моделируется как `Binomial(accepted, pd_annual)`;
- портфельный payoff считается как:
  - прибыль на принятие: `40`;
  - убыток на дефолт: `LGD * EAD`.

3. Затем рассчитываются агрегаты распределения:
- `npv_mean`, `npv_std`;
- `npv_p05 / npv_p50 / npv_p95`;
- `var95`, `cvar95`;
- среднее число принятий, дефолтов и использование капитала.

### 2.3 Sensitivity-анализ

Дополнительно выполнен grid-test для `base`-сценария:
- `p_accept`: `0.90x / 1.00x / 1.10x`;
- `PD`: `0.90x / 1.00x / 1.10x`.

Это нужно, чтобы понять, какие драйверы сильнее двигают портфельный результат и downside.

## 4) Что интересного в sensitivity

Таблица [data/simulation_sensitivity.csv](data/simulation_sensitivity.csv) показывает:

1. Рост `PD` ухудшает результат очень сильно и почти линейно.
2. Рост `p_accept` в текущей постановке тоже ухудшает `npv_mean`.
3. Это важный сигнал: при заданной формуле payoff дополнительное принятие оффера несет больше ожидаемого кредитного убытка, чем прибыли.

Наиболее "мягкая" комбинация в сетке:
- `p_accept = 0.9x`, `PD = 0.9x`,
- `npv_mean ≈ -560,200`.

Наиболее тяжелая комбинация:
- `p_accept = 1.1x`, `PD = 1.1x`,
- `npv_mean ≈ -937,008`,
- `CVaR95 ≈ -997,163`.

## 5) Насколько это соответствует заданию

По сути задания Notebook 05 закрывает важную часть требований:
- есть Монте-Карло симуляция;
- есть вероятностная оценка риск-экспозиции;
- есть стрессирование сценариев;
- есть результаты симуляций для стратегии, полученной на предыдущем шаге.

Но есть и ограничения относительно полного идеала из assessment:

1. Симуляция пока одношаговая, а не полноценная многопериодная модель жизненного цикла займа.
2. Внутри симуляции оценивается одна рекомендованная стратегия (`lp_round`), а не полное сравнение нескольких стратегий в Монте-Карло.
3. Формула payoff сейчас более консервативна, чем логика `expected_npv` в Notebook 04.

Это означает:
- технически блок выполнен корректно;
- методологически он полезен;
- но для полной согласованности с assignment модель симуляции еще можно усилить.

## 6) Ключевой нюанс: расхождение с Notebook 04

Это главный вывод всего Notebook 05.

В Notebook 04 детерминированный `expected_npv` для `lp_round` положительный:
- `base`: `322,500.91`
- `stress`: `285,584.34`
- `benign`: `341,399.07`

В Notebook 05 simulated `npv_mean` во всех сценариях отрицательный.

Причина не в технической ошибке исполнения, а в различии экономической логики:
- в Notebook 04 используется агрегированный `ev_per_increase` из риск-модуля;
- в Notebook 05 используется явная формула `40 - default * LGD * EAD`.

Если при текущих `PD`, `LGD` и `EAD` ожидаемый убыток на дефолт превышает фиксированные `40`, то увеличение `p_accept` ухудшает портфельную экономику.

Это делает Notebook 05 особенно ценным: он не просто подтверждает результат оптимизации, а проверяет, не слишком ли оптимистична детерминированная постановка.

## 7) Практический вывод

На текущей конфигурации данных и допущений:
- стратегия исполняется технически корректно;
- сценарный ranking интуитивен (`benign > base > stress`);
- downside-риски измерены и хорошо читаются;
- но экономическая модель симуляции показывает отрицательный NPV во всех сценариях.

Значит, перед финальной рекомендацией бизнесу нужно сделать одно из двух:

1. Либо согласовать Notebook 04 и Notebook 05 на одной и той же unit-economics логике.
2. Либо прямо зафиксировать, что симуляция является более консервативной валидацией и указывает на риск переоценки доходности в детерминированном оптимизаторе.
