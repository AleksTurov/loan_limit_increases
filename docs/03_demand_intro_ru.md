# Intro к ноутбуку 03 — Прогноз спроса и макро-сценарии

## 1) Цель ноутбука

Ноутбук [notebooks/03_Demand_ru.ipynb](notebooks/03_Demand_ru.ipynb) решает задачу оценки спроса на увеличение лимита:

- оценить вероятность принятия оффера `p_accept` для каждого клиента;
- учесть влияние макроусловий через сценарии `base / stress / benign`;
- подготовить таблицу для оптимизатора (Notebook 04).

Выходной артефакт ноутбука: [data/df_demand.csv](data/df_demand.csv).

## 2) Что было сделано

### 2.1 Входные данные

Использован файл [data/df_risk.csv](data/df_risk.csv), полученный в Notebook 02.

Ключевые поля на входе:
- клиентские признаки (`initial_loan`, `days_since_last_loan`, `on_time_pct`, `risk_label`);
- риск-параметры (`pd_monthly`, `pd_annual`, `ev_per_increase`);
- технические признаки (`utilisation_rate`, `max_possible_increases`);
- индикаторы `eligible` и `received`.

### 2.2 Проблема с таргетом и принятый прокси

В исходном датасете нет явных событий:
- `offer_shown`;
- `offer_accepted`.

Поэтому использован прокси-таргет:
- оставляем только `eligible == 1`;
- считаем `received` как `accepted_proxy`.

Это практическая аппроксимация спроса для оптимизационной постановки, а не чистая причинная оценка реакции клиента.

### 2.3 Модель спроса

Реализовано в [src/demand_model.py](src/demand_model.py):

1. Базовый кандидат: логистическая регрессия по признакам:
- `risk_label`
- `initial_loan`
- `days_since_last_loan`
- `on_time_pct`
- `utilisation_rate`
- `max_possible_increases`

2. Контроль качества на holdout через ROC-AUC.

3. Защитный режим при слабом сигнале:
- если `AUC < 0.53`, включается `segment_fallback`;
- базовая вероятность берется из эмпирических tier-priors;
- добавляются мягкие ограниченные корректировки по `days_since_last_loan` и `utilisation_rate`;
- итог ограничивается в диапазоне `[0.01, 0.99]`.

Это сделано, чтобы не переинтерпретировать шум как предиктивный сигнал.

### 2.4 Макро-сценарии

В [src/demand_model.py](src/demand_model.py) заданы три сценария:

- `base`: нейтральная среда;
- `stress`: снижение вероятности принятия, сильнее для более рискованных tier;
- `benign`: рост вероятности принятия, сильнее для более рискованных tier.

Сценарные вероятности строятся из `p_accept_base` через tier-мультипликаторы и глобальный сдвиг.

## 4) Почему это методологически корректно

1. В EDA уже показано, что исторические назначения во многом случайны, поэтому высокий предиктивный сигнал по `received` не ожидается.
2. При `AUC ~ 0.5` использовать "чистую ML-модель" как основу бизнес-решений рискованно.
3. Fallback-подход дает:
- устойчивость;
- прозрачность;
- контролируемую связь с бизнес-допущениями.
4. Сценарный блок позволяет прямо проверять влияние макроусловий на спрос, как требует задание.

## 5) Основные выводы по Notebook 03

1. Выделить устойчивый индивидуальный ML-сигнал принятия оффера на текущих данных не удалось.
2. Для задачи оптимизации более надежен сегментно-сценарный подход к `p_accept`.
3. Макроусловия оказывают существенное влияние на ожидаемое число принятий.
4. Результат Notebook 03 готов как вход в Notebook 04: можно оптимизировать аллокацию с учетом
- `p_accept_scenario`;
- `ev_per_increase`;
- бизнес-ограничений.

## 6) Ограничения и что улучшать дальше

1. Прокси-таргет (`received`) не равен реальному факту принятия оффера.
2. Нет временных логов оффера/клика/принятия — это ограничивает causal-интерпретацию спроса.
3. Сценарные множители пока экспертные; при наличии внешних рядов их нужно калибровать статистически.
4. В следующем шаге (Notebook 04) важно делать чувствительность решения к `p_accept`, а не к одной фиксированной оценке.
