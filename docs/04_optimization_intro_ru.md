# Notebook 04: Оптимизация портфеля (RU)

## 1) Цели

В Notebook 04 решается задача аллокации повышений лимита при ограниченном капитале.

Основные цели:
- максимизировать ожидаемый NPV портфеля;
- сравнить 3 стратегии: lp_relax, lp_round, greedy;
- проверить устойчивость решения в макро-сценариях base, stress, benign;
- подготовить выходные таблицы для следующего шага моделирования.

Связанные файлы:
- [notebooks/04_Optimization_ru.ipynb](notebooks/04_Optimization_ru.ipynb)
- [notebooks/04_Optimization.ipynb](notebooks/04_Optimization.ipynb)
- [src/optimizer.py](src/optimizer.py)

## 2) Что делали

### 2.1 Подготовка входа

На вход берется [data/df_demand.csv](data/df_demand.csv) из Notebook 03.

Так как в этом файле не всегда есть все поля для оптимизатора, добавлены fallback-правила:
- если нет max_possible_increases, подставляется 1;
- если нет ead, подставляется initial_loan.

Это реализовано в [src/optimizer.py](src/optimizer.py) и продублировано в setup-ячейках обоих ноутбуков для прозрачности.

### 2.2 Постановка оптимизации

Переменная решения:
- x_i = число предложенных увеличений лимита клиенту i.

Целевая функция:
- максимизация суммы ожидаемых вкладов в NPV по клиентам.

Ограничения:
- 0 <= x_i <= eligible_i * max_possible_increases_i;
- суммарная ожидаемая нагрузка на капитал <= 30% от суммарного initial_loan.

### 2.3 Сравнение стратегий

Считались и сравнивались:
- lp_relax: LP-релаксация (нецелочисленная верхняя граница);
- lp_round: округление LP с дозаполнением по остаточному капиталу;
- greedy: целочисленный жадный алгоритм по плотности ценности.

Добавлены sanity checks:
- capital_usage_pct <= 100%;
- нет отрицательных аллокаций;
- expected_npv для lp_relax не ниже лучших integer-стратегий.

Практический вывод:

- капитал везде используется почти полностью, но в рамках ограничения.

## 4) Нюансы

1. Важная техническая деталь: схема данных между этапами может отличаться, поэтому fallback-поля обязательны для стабильного прогона.
2. При текущих параметрах задачи капитал является почти полностью связывающим ограничением, поэтому стратегии сходятся по результату.
3. Стратегия lp_relax не может быть напрямую применена в проде, но полезна как верхняя оценка качества.
4. Для бизнес-интерпретации следует смотреть не только NPV, но и сегментный микс (mix_prime, mix_near, mix_sub) по сценариям.
5. Если изменятся p_accept или EV-параметры, рекомендуется повторять сравнение стратегий и чувствительность к capital_limit_ratio.

## 6) Что здесь действительно интересного

1. Очень маленький integrality gap:
  - в каждом сценарии разница между lp_relax и лучшей целочисленной стратегией составляет около 0.14-2.60 NPV,
  - это практически ноль на масштабе портфеля (сотни тысяч NPV).

2. Сильная чувствительность к макро-сценарию:
  - stress vs base по integer-решению: примерно -11.45% по expected_npv,
  - benign vs base: примерно +5.86% по expected_npv.

3. При stress модель предлагает больше клиентов (около +12.1% к base), но с меньшей ценностью на единицу аллокации:
  - offered_customers растет,
  - expected_npv снижается.

4. Сегментный сдвиг портфеля по сценариям:
  - в stress падает доля Prime (около -3.60 п.п.) и растет Near-Prime (около +3.62 п.п.),
  - это индикатор того, что оптимум перераспределяется в менее качественный микс при ухудшении среды.

5. Капитал является жестко связывающим ограничением во всех сценариях:
  - capital_usage_pct почти 100%,
  - значит, именно управление капиталом определяет форму оптимального решения.

## 7) Что можно добавить сильного

1. Робастная постановка для решения “на один план”:
  - оптимизировать не только base, а max(min(NPV_base, NPV_stress, NPV_benign))
  - или max(ожидаемый NPV - lambda * downside).

2. Эффективная граница по капиталу:
  - прогонять capital_limit_ratio по сетке (например 10%-60%),
  - строить кривую "капитал -> NPV" и выбирать рабочую точку с учетом риск-аппетита.

3. Риск-ограничения по миксу:
  - добавить явные caps/floors по сегментам (например max доля Sub-Prime),
  - это повышает управляемость решения и делает его ближе к кредитной политике.

4. Ограничения концентрации:
  - лимиты по группам клиентов (tier, vintages, продуктовые корзины),
  - чтобы избежать скрытой концентрации в “узких” участках портфеля.

5. Более бизнесовый ranking стратегий:
  - кроме expected_npv, добавить сравнение по NPV/капитал, NPV/клиент, downside в stress,
  - и уже по набору KPI выбирать рекомендованную стратегию.

6. Валидация стабильности аллокации:
  - небольшой bootstrap/perturbation входных p_accept и ev_per_increase,
  - оценка, насколько ранжирование стратегий и клиентские рекомендации устойчивы к шуму.
